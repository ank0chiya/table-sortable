# 開発・実装ガイドライン (AIアシスタント共通ルール)

本プロジェクトにおいて、機能実装、リファクタリング、バグ修正を行う際は、システムを安定的かつ拡張性高く保つため、以下のプロフェッショナルおよびエンタープライズ水準の開発規約を必ず遵守すること。

## 1. アーキテクチャ設計 (ドメイン駆動設計: DDD のレイヤードアーキテクチャ)

ビジネスロジックの散逸を防ぎ、テスト容易性と保守性を高めるため、以下の層（レイヤー）を明確に分離して実装する。

- **Domain 層 (ドメイン層)**
  - 最重要のレイヤー。ビジネスのルールや型定義（Entities, Value Objects）を配置する。
  - **ルール**: UI（React）や外部API、ライブラリなど、他のいかなる層・技術にも依存してはならない。
- **Application (Use Case) 層**
  - アプリケーションが提供する機能（ユースケース）の処理フローを管理する。
  - **ルール**: 具体的なUI操作や外部通信の実装はもたず、Domain層の操作とInfrastructure層の呼び出しを統制する（カスタムフックやサービス関数として実装）。
- **Infrastructure 層 (インフラストラクチャ層)**
  - 外部システムとのやり取り（API Fetch、Local Storage連携など）を担当する。
  - **ルール**: Application層で定義されたインターフェース（Repositoryなど）を実装する形で依存性を逆転させる（DIP: 依存性逆転の原則）。
- **Presentation / UI 層**
  - ユーザーからの入力を受け付け、データを表示する（Next.js pages/app, React Components）。
  - **ルール**: ビジネスロジックを含めない。状態の表示、ユーザーイベントの発火、およびApplication層のフック呼び出しに専念する。

## 2. コンポーネント設計とUI実装

- **関心の分離 (Container / Presentational)**
  - 状態管理やデータ取得を担う Container コンポーネントと、純粋にUI描画のみを行う Presentational コンポーネントを分離し、コンポーネントの再利用性を高める。
- **単一責任の原則 (SRP)**
  - 1つのコンポーネント・関数は「1つの役割」のみを持つ。肥大化した関数やコンポーネントは適切に分割すること。

## 3. 型安全性と厳格なコーディング規約

- **Strict TypeScript**
  - `strict` モードを常に前提とする。
  - `any` の使用、および無根拠な型アサーション (`as`) は原則禁止。型が不明な場合は `unknown` を用い、ランタイムでの型アサーションバリデーション（Type Guard）を実装する。
- **境界でのデータバリデーション**
  - 外部 API からのデータやユーザーフォームからの入力など「境界」を越えるデータには、Zod等のバリデーションライブラリを用いてランタイムにおける安全性を担保する。
- **マジックナンバーとハードコードの排除**
  - 意味を持つ数値や文字列は各種定数ファイル (`constants.ts` 等) で一元管理する。

## 4. エラーハンドリングと堅牢性 (Robustness)

- **予期せぬエラーとビジネスエラーの分離**
  - システムクラッシュに繋がる「予期せぬエラー」と、ユーザーの入力不備等の「想定されたビジネスエラー」を明確に区別し、ユースケースごとに適切なユーザーフィードバック（エラー表示）を行う。
- **グローバルエラーバウンダリ**
  - アプリケーション全体がクラッシュしてホワイトスクリーンになることを防ぐため、React Error Boundaryを適切な粒度で配置し、フェイルセーフな設計とする。

## 5. テスト戦略と品質保証

- **自動テストの徹底**
  - ビジネスロジック（Domain層、Application層）は単体テスト (Unit Test) で品質を担保する。
  - 必要に応じて、主要なユーザーフローについては E2E テスト・統合テストを考慮する。

## 6. パフォーマンス最適化

- **不要な再レンダリングの抑制**
  - `React.memo`, `useMemo`, `useCallback` を効果的に用いる。ただし無秩序な最適化は避け、明確なプロファイリング根拠に従う。
- **非同期処理の適切なフィードバック**
  - ロード中のスケルトン表示やスピナー、エラー時の再試行（Retry）UIなどを欠かさず実装し、常にユーザーのシステムに対する信頼感を確保する。

## AIへの指示 (プロンプト) としての留意点
AIアシスタントはコードを生成・修正する際、必ず本 `GEMINI.md` の原則に合致しているかを自己検証し、反している場合は適切な設計へと是正したうえで提案を行うこと。
